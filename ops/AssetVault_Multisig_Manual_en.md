# AssetVault Production Operations Manual (Draft)

This document is for production operators, signers, and reviewers. It covers:

- `AssetVault` governance structure
- What each role can do
- Pre-execution security checks
- How to fill transactions in the Safe web UI
- Lark approval requirements

`ADMIN_ROLE` and `UPGRADE_ROLE` must go through Timelock. Other privileged actions are direct Safe-to-`Vault Proxy` paths.

# Governance Structure

Recommended setup:

- `Governance Safe (4/7)` -> `Governance Timelock (72h)` -> `UPGRADE_ROLE`
- `Governance Safe (4/7)` -> `DEFAULT_ADMIN_ROLE`
- `Admin Safe (4/7)` -> `Admin Timelock (72h)` -> `ADMIN_ROLE`
- `Token Safe (4/7)` -> `TOKEN_ROLE`
- `Validator Safe (4/7)` -> `VALIDATOR_ROLE`
- `Emergency Guardian Safe (3/5)` -> `PAUSE_ROLE`
- `Operator KMS` -> `OPERATOR_ROLE`
- `Deposit Service Wallet` -> `DEPOSIT_ROLE`

Key points:

- Only `UPGRADE_ROLE` is used for contract upgrades, and it must be timelocked.
- `ADMIN_ROLE` controls challenge period, rebalance receiver, and fee withdrawal, and it must also be timelocked.
- `DEFAULT_ADMIN_ROLE` is not used for upgrades.
- `TOKEN_ROLE`, `VALIDATOR_ROLE`, and `PAUSE_ROLE` are direct Safe-to-`Vault Proxy` paths.
- For validator rotation, always `addValidators` first, confirm the new set works, then `removeValidators` the old set.

# Security Requirements

## Core Rules

- All production actions must go through Safe or Safe + Timelock. No production EOA should call privileged functions directly.
- No blind signing. Every signer must independently verify the UI parameters and the calldata generated by Safe.
- Verify Safe domain, chain ID, Safe address, Vault Proxy address, Timelock address, and receiver address before signing.
- Production signing must be performed on dedicated devices with hardware wallets.
- New addresses, tokens, validators, and implementations must be cross-checked from trusted sources.
- Every executed action must be verified on-chain. A successful transaction status alone is not sufficient.

## Pre-Execution Checks

- Access Safe only via bookmarks or manual URL entry. The domain should be `https://app.safe.global`.
- Check the browser certificate and avoid signing on lookalike sites.
- Confirm the connected wallet is the intended hardware wallet.
- Confirm the chain and Safe address shown in the Safe UI are correct.
- Confirm whether `To` should be `Vault Proxy`, `Admin Timelock`, or `Governance Timelock`.
- For any new receiver / treasury / validator / implementation address, verify from at least two independent sources:
  - Official website, docs, GitHub, or announcement
  - Verified contract page on a block explorer
  - Internal address registry
- For implementation addresses, confirm:
  - the implementation contract has been audited
  - bytecode exists at the target address
  - source is verified or traceable
  - storage layout compatibility has been reviewed
- For token addresses, confirm:
  - correct chain
  - correct decimals
  - official token, not a fake with the same symbol

## Additional Operational Rules

- Do not send “new addresses” in chat without traceable source and purpose.
- For a new receiver, prefer a small-value test before first large transfer.
- Do not use unnecessary browser extensions on production machines.
- Do not ask others to review raw calldata without also providing function signature and parameter meaning.
- Review signer lists, rebalance receiver settings, validator sets, and address registry at least monthly.

## Lark Approval

- All privileged production actions must go through a Lark approval process before Safe or Safe + Timelock execution.
- Approval records should include at least: operation name, target chain, target contract, target function, parameters, business reason, risk notes, and transaction hash.
- Timelocked actions should also record: `salt`, `schedule` transaction hash, and `execute` transaction hash.
- Each Lark approval record must map cleanly to the final on-chain transaction.

# Operations

This section assumes operators use the `Multisig Calldata Checker`. Operators should not need `cast` or any other CLI tooling.

- Tool URL: `https://web-three-mauve-40.vercel.app`

## How To Use The Multisig Calldata Checker

### Direct Actions

1. Select the chain.
2. Select the target operation.
3. In `Encode` mode, fill in the parameters.
4. The tool will show:
   - the correct `To`
   - the correct `ABI JSON`
   - the generated calldata
5. In Safe `Transaction Builder`, fill the same `To`, `ABI JSON`, and parameters.
6. On the Safe `Review` page, copy the generated `Data`.
7. Return to the checker in `Decode` mode, paste the `Data`, and confirm the decoded function and parameters exactly match the approved request.
8. Sign only when the checker `Encode` result, Safe `Review` `Data`, and checker `Decode` result are all consistent.

### Timelocked Actions

`ADMIN_ROLE` and `UPGRADE_ROLE` operations are timelocked and must be handled as two layers of calldata.

1. In the checker, select the target operation, fill the business parameters, and generate the inner calldata.
2. Confirm the inner calldata `To` is `Vault Proxy`.
3. Then generate the Timelock outer calldata for:
   - `schedule`
   - `execute`
4. Confirm the outer calldata `To` is the operation-specific Timelock, not `Vault Proxy`.
5. For both `schedule` and `execute`, verify:
   - `target` is `Vault Proxy`
   - `data` exactly matches the inner calldata
   - `predecessor` is correct
   - `salt` matches the approval record
   - `delay` is the expected value
6. On the Safe `Review` page, copy the outer `Data` and decode it again in the checker.
7. `execute` must use the exact same `target / data / predecessor / salt` as `schedule`. The only extra field in `schedule` is `delay`.

## General Review Rules

- Do not hand-write raw calldata. Use the checker output as the source of truth.
- The Safe parameters, checker `Encode` result, Safe `Review` `Data`, and checker `Decode` result must all match.
- For address parameters, also confirm the business meaning of the address: treasury, receiver, validator, or implementation.
- For arrays and validator sets, verify every item, including order and numeric values.
- For Timelock actions, keep the inner calldata, `schedule` tx hash, `execute` tx hash, and `salt` in the approval record.

## Supported Actions

### grantRole

- Required Role: `DEFAULT_ADMIN_ROLE`
- Path: `Governance Safe -> Vault Proxy`
- Parameters:
  - `role`: role hash to grant
  - `account`: recipient address
- Review focus:
  - confirm the selected `role` is correct
  - confirm `account` matches the approval record

### revokeRole

- Required Role: `DEFAULT_ADMIN_ROLE`
- Path: `Governance Safe -> Vault Proxy`
- Parameters:
  - `role`: role hash to revoke
  - `account`: target address
- Review focus:
  - do not revoke a signer or service account that is still in use

### upgradeToAndCall

- Required Role: `UPGRADE_ROLE`
- Path: `Governance Safe -> Governance Timelock -> Vault Proxy`
- Parameters:
  - `newImplementation`: new implementation address
  - `data`: migration or initialization calldata, or `0x`
- Review focus:
  - `newImplementation` must be the approved audited production implementation
  - `data` must be explicitly understood and approved
  - the outer `schedule` and `execute` calldata must both correspond to the same inner calldata

### updatePendingWithdrawChallengePeriod

- Required Role: `ADMIN_ROLE`
- Path: `Admin Safe -> Admin Timelock -> Vault Proxy`
- Parameters:
  - `newValue`: new challenge period in seconds

### setRebalanceReceiver

- Required Role: `ADMIN_ROLE`
- Path: `Admin Safe -> Admin Timelock -> Vault Proxy`
- Parameters:
  - `newReceiver`: new rebalance receiver
- Review focus:
  - `newReceiver` must come from the internal address registry

### withdrawFees

- Required Role: `ADMIN_ROLE`
- Path: `Admin Safe -> Admin Timelock -> Vault Proxy`
- Parameters:
  - `tokens`: token address array, use `address(0)` for native token
  - `to`: fee receiver
- Review focus:
  - `to` must be the approved treasury or designated receiver

### addToken

- Required Role: `TOKEN_ROLE`
- Path: `Token Safe -> Vault Proxy`
- Parameters:
  - `token`: new token address, use `address(0)` for native token
  - `hardCapRatioBps`: fast-withdraw ratio
  - `refillRateMps`: refill rate
- Review focus:
  - confirm chain, token address, decimals, and token ownership/source

### updateToken

- Required Role: `TOKEN_ROLE`
- Path: `Token Safe -> Vault Proxy`
- Parameters:
  - `token`: supported token address
  - `hardCapRatioBps`: new fast-withdraw ratio
  - `refillRateMps`: new refill rate

### addValidators

- Required Role: `VALIDATOR_ROLE`
- Path: `Validator Safe -> Vault Proxy`
- Parameters:
  - `validators`: validator array with `signer` and `power`
  - `requiredPower`: approval threshold
- Review focus:
  - validator order must be correct
  - `power` and `requiredPower` must be checked item by item

### updateValidatorRequiredPower

- Required Role: `VALIDATOR_ROLE`
- Path: `Validator Safe -> Vault Proxy`
- Parameters:
  - `validators`: must exactly match the current on-chain set
  - `newRequiredPower`: new threshold

### removeValidators

- Required Role: `VALIDATOR_ROLE`
- Path: `Validator Safe -> Vault Proxy`
- Parameters:
  - `validators`: validator set to remove, must exactly match the on-chain set
- Review focus:
  - confirm the new validator set is already active
  - confirm signing services have already switched

### toggle

- Required Role: `PAUSE_ROLE`
- Path: `Emergency Guardian Safe -> Vault Proxy`
- Parameters:
  - `pause`: `true` to pause, `false` to unpause
